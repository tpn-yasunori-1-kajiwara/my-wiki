# Wiki編集ルール

## 概要
本リポジトリの `CLAUDE.md` に集約された、AI（Claude Code）が `wiki/` を編集する際の運用ルール一式。[[3層構造]] の **schema 層** にあたる。フロントマター仕様、URL の扱い、コマンド・コードブロックの保存方針、`/ingest` の判断手順など、AI に守らせたい規約を成文化することで、編集行動の暴走と一貫性崩れを防ぐ。

## 本文

### 重要な原則

- `raw/` のファイルは **絶対に削除・改名・上書きしない**（不変層）。
- `wiki/` の各ページは、出典として `raw/` のファイル名を `sources:` フロントマターに必ず記録する。これが **「処理済みマーカー」** を兼ねる。
- 同じ raw ファイルの内容を二重に wiki に登録しない（既存ページがあれば追記）。
- wiki ページは **テキストのみ**。画像埋め込み（`![](...)`）は使わない。

### wiki/ ページの構造（フロントマターテンプレート）

```markdown
---
title: ページタイトル（人間可読・日本語可）
aliases: [別名1, 別名2]                # 任意
tags: [タグ1, タグ2]                    # kebab-case 推奨
sources:                                # 出典 raw ファイル。複数可。ファイル名のみ
  - "2026-05-12_data-pipeline.md"
references:                             # 任意。WebFetchで参照したURL
  - "https://example.com/article"
created: 2026-05-12
updated: 2026-05-12
---

# ページタイトル

## 概要
（1〜3行の要約）

## 本文
（概念の詳しい説明。必要に応じて WebFetch で取得した参考情報を反映）

## 関連
- [[他のページタイトル]]
```

### フロントマターのルール

- `sources` は必須（meta タグページを除く）。raw のファイル名のみを記載。
- `references` は任意。WebFetch で参照した URL を列挙。
- `updated` は AI が編集するたびに今日の日付に更新する。
- `created` は最初の作成日。上書きしない。
- `tags` は kebab-case 推奨。

### URL が含まれる場合の取り扱い

raw/ のテキストに URL があれば、AI は **WebFetch** で取得し内容を wiki に反映する。WebFetch の `prompt:` には、要約だけでなく **「コマンド・コードブロック・設定例・コマンドラインオプション・JSON/YAML/シェルスニペットを原文ママで抽出すること」** を明示的に依頼する。

- 取得した URL は `references:` に記録する（オプション）。
- 認証必要・取得失敗の URL は本文に「不明: URL取得失敗（理由）」と明示する。
- 無限再帰や明らかに大量取得になる場合は数件で切り上げる。

### コマンド・コードブロック・設定例の保存（重要）

raw/ 本文や WebFetch 結果に **コマンド・コードブロック・設定例**（CLI 例、シェル、`settings.json` 等の設定値、JSON/YAML、API 呼び出し例、コードスニペット、関数シグネチャ、ファイルパス、フラグ名）が含まれる場合、それらは **要約せず原文（または最小限の整形）でコードブロックとして wiki に保存する**。

- 「概念名・キーワード名」だけ書いて実体のコマンド・設定値を省略するのは禁止（後から「コピペできる資料」として参照したいため）。
- コードブロックには言語タグを付ける（例: ` ```json `, ` ```bash `, ` ```yaml `）。
- 引用元での文脈や注意書きがあれば、コードブロックの直前または直後に短く添える。
- 量が多すぎてページが膨れる場合は、最も実用的なコマンド／設定を残し、残りは `references:` の URL に委ねる旨を本文で明示する。

**良い例:**

````markdown
### PreToolUse フックの設定例

```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "command": "/path/to/check.sh" }
    ]
  }
}
```
````

**悪い例:**

```markdown
PreToolUse フックは settings.json で設定できる。
```
（← 何をどこにどう書けばよいかが復元できない）

### 粒度ルール

- 1 つの raw ファイル → 複数の wiki ページが基本。
- 1 ページの目安: 200〜800 字程度。
- 極端に短い断片は独立ページにせず、関連ページに統合する。

### ファイル名規約

- `raw/` の推奨命名: `YYYY-MM-DD_短いスラッグ.md`（例: `2026-05-12_data-pipeline.md`）。日本語ファイル名も可。
- Windows 禁止文字（`\ / : * ? " < > |`）は使わない。
- `wiki/` の `[[他のページ名]]` リンクは **ファイル名（拡張子なし）と一致** させる。

### /ingest 時の判断手順

1. **対象列挙**: `raw/` を Glob で取得（`.gitkeep` は除外）。
2. **既処理判定**: 各ファイル名について `wiki/**/*.md` を Grep し、`sources:` に記載があればスキップ。
3. **読込**: Read ツールで未処理ファイルを読む。
4. **URL 抽出**: 本文中の URL を WebFetch で取得。prompt には「コマンド・コードブロック・設定例を原文ママで抽出してほしい」旨を含める。
5. **概念抽出 + コマンド抽出**: テキスト + URL 内容から主要な概念と、属するコマンド・設定例を列挙。
6. **既存ページ照合**: 各概念について `wiki/` の title/aliases/ファイル名・全文を Grep。
7. **編集**:
   - 既存ページあり → Edit で追記、`sources:` 追加、`updated:` 更新、必要なら `references:` 追加。該当ページにコマンド・設定例がまだ載っていなければ合わせて追記。
   - なし → Write で新規作成。抽出したコマンド・設定例は本文中にコードブロックで保存する。
   - 矛盾発見 → `> **要レビュー:** ...` フラグを残す（自分で解決しない）。
8. **相互リンク**: 新規/更新ページから関連既存ページへ `[[link]]`、可能なら逆方向も。
9. **報告**: 処理結果を簡潔にまとめる。**raw/ のファイルは絶対に削除・改名しない**。

### やってはいけないこと

- `raw/` のファイルの削除・改名・編集
- 画像を埋め込むこと（`![](...)`）
- `sources:` 未記録のまま wiki に書くこと（処理済み判定が壊れる）
- WebFetch なしで「URL の内容はおそらく〜」と憶測で書くこと
- 既存ページの全面書き換え（追記・修正で対応）
- フロントマターの省略
- 確証のない断定（不明点は「不明: 〜」と明示）
- **コマンド・コードブロック・設定例を「概念名だけ」に要約して落とすこと**（実体が復元できなくなる）

### /lint で検出すべき項目

- `sources:` が空のページ（meta タグ付きページは除外）
- `sources:` 記載のファイルが `raw/` に存在しないページ（dead source）
- どこからもリンクされていない孤立ページ
- title 重複 / aliases 重複
- 「要レビュー」フラグが残っているページ
- `raw/` にあるが、どの wiki ページの `sources:` にも記載がないファイル（処理漏れ）
- wiki ページ内の画像埋め込み `![](...)`（仕様違反）
- 設定キー名・コマンド名（`denyRead` / `PreToolUse` / `gcloud auth login` 等）が本文に登場するのに、対応するコードブロックが 1 つもないページ（コマンド省略の疑い）
