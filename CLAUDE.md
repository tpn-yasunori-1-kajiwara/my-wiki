# 個人用 LLM Wiki — 運用ルール（schema層）

このリポジトリは、図解（PNG）を一次資料として個人用の知識ベース（wiki）を育てるためのものです。
AI（Claude Code）は本ファイルのルールに従って `wiki/` を編集します。

## レイヤー構成

| パス        | 役割                                                     | 編集主体      |
|-------------|----------------------------------------------------------|---------------|
| `raw/`      | 一次資料（PNG図解）。**不変（immutable）**として扱う     | 人間          |
| `wiki/`     | 概念ごとに整理された Markdown 知識ベース                 | AI（主）/人間 |
| `CLAUDE.md` | AIの振る舞いルール（このファイル）                       | 人間          |

重要な原則:
- `raw/` のファイルは**絶対に削除・改名・上書きしない**。原典として保つ。
- `wiki/` の各ページは、出典として `raw/` のPNGファイル名をフロントマターに必ず記録する。
- 同じPNGの内容を二重に登録しない（既存ページがあれば追記）。

## raw/ の命名規約

- 推奨: `YYYY-MM-DD_短い英語スラッグ.png`（例: `2026-05-12_data-pipeline.png`）
- 日本語ファイル名も許容するが、空白を含めない（`_` または `-` で区切る）
- 上記に従っていないファイルもそのまま受け入れる。AI側でファイル名から内容を推測しないこと。**内容は必ず画像を読んで判断する**。

## wiki/ ページの構造

各 Markdown ファイルは以下のフロントマターを持つ。

```markdown
---
title: ページタイトル（人間可読・日本語可）
aliases: [別名1, 別名2]      # 検索や [[link]] の揺れ対策。任意
tags: [タグ1, タグ2]          # 横断検索用。kebab-case 推奨
sources:                      # 出典PNG。複数可。raw/からの相対パス
  - 2026-05-12_data-pipeline.png
created: 2026-05-12
updated: 2026-05-12
---

# ページタイトル

## 概要
（1〜3行の要約）

## 本文
（概念の詳しい説明）

## 関連
- [[他のページタイトル]]
- [[さらに別のページ]]

## 出典
![](../raw/2026-05-12_data-pipeline.png)
```

### フロントマターのルール
- `sources` は必須。出典PNGを必ず1つ以上記載する。
- `updated` はAIが編集するたびに更新する（今日の日付）。
- `created` は最初の作成日。上書きしない。
- `tags` は名詞句、kebab-case（例: `data-pipeline`, `auth-flow`）。日本語タグも可だが英語を優先。

## 粒度ルール

**1枚のPNG = 複数のwikiページ** が基本。
- PNGに登場する概念ごとにページを分けること。
- 1ページの目安: A4で1〜2ページ相当（200〜800字程度）。長すぎる場合は分割する。
- ただし、極端に短い断片（1〜2文）の独立ページは作らず、関連ページに統合する。

## ファイル名規約（wiki/）

- ファイル名は **日本語可**、ただしWindows禁止文字（`\ / : * ? " < > |`）は使わない。
- 推奨例: `wiki/データパイプライン.md`, `wiki/認証フロー.md`
- `[[他のページ名]]` リンクは **ファイル名（拡張子なし）と一致** させる。
  - Obsidianとの互換性のためこの形式を採用。

## リンクのルール

- ページ間リンクは Obsidian 形式 `[[ページ名]]` を使用。
- 関連性が薄い場合はリンクしない。**孤立ページは許容**（あとで /lint で検出）。
- 画像参照は `![](../raw/ファイル名.png)` のMarkdown形式（Obsidianも解釈可）。

## 更新方針（/ingest 時の判断基準）

新しいPNGを処理するとき:

1. **概念抽出**: PNGを画像として読み、登場する主要な概念を列挙する。
2. **既存ページの照合**: 各概念について `wiki/` を検索（title, aliases, ファイル名で照合）。
3. **判断**:
   - **既存ページに該当する** → 本文に追記し、`sources:` にPNGファイル名を追加、`updated:` を今日に更新。
   - **既存ページにない** → 新規ページを作成。
   - **既存ページの内容と矛盾する** → 自分で解決せず、ページ末尾に `> **要レビュー:** 既存記述と新出典の矛盾点を箇条書き` という形でフラグを残す。
4. **相互リンク**: 新規/更新したページから、関連する既存ページへ `[[link]]` を張る。逆方向の追記もできる範囲で行う。
5. **記録**: 処理完了したPNGについて、対応するwikiページに `sources:` で参照を残すことで「処理済み」を示す。別途のジャーナルは作らない。

## やってはいけないこと

- `raw/` 内のファイルの削除、改名、編集
- 画像を見ずに、ファイル名だけで内容を推測してページを作ること
- 既存ページの全面書き換え（追記・修正・分割で対応する）
- フロントマターの省略
- 確証のない断定を書くこと（不明点は本文に「不明: XXX」と明示する）

## /lint で検出すべき項目

- `sources:` が空のページ
- 出典PNGが `raw/` に存在しないページ（dead source）
- どこからもリンクされていないページ（孤立）
- title重複 / aliases重複
- 同じ概念を扱う複数ページ（疑い）
- 「要レビュー」フラグが残っているページ
