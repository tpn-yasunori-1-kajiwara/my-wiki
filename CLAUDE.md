# 個人用 LLM Wiki — 運用ルール（schema層）

このリポジトリは、テキスト資料を一次資料として個人用の知識ベース（wiki）を育てるためのものです。
AI（Claude Code）は本ファイルのルールに従って `wiki/` を編集します。

## レイヤー構成

| パス        | 役割                                                                              | 編集主体        |
|-------------|-----------------------------------------------------------------------------------|-----------------|
| `raw/`      | 一次資料（テキスト）。**不変（immutable）**。記事メモ・要約・クリッピング等        | 人間（投入のみ）|
| `wiki/`     | 概念ごとに整理された Markdown 知識ベース                                          | AI（主）/人間   |
| `scripts/`  | 自動化スクリプト（定期ポーリング等）                                              | 人間            |
| `CLAUDE.md` | AIの振る舞いルール（このファイル）                                                | 人間            |

### 重要な原則
- `raw/` のファイルは **絶対に削除・改名・上書きしない**（不変層）。
- `wiki/` の各ページは、出典として `raw/` のファイル名を `sources:` フロントマターに必ず記録する。
  - これが **「処理済みマーカー」** を兼ねる。同名が記録されていれば再処理しない。
- 同じ raw ファイルの内容を二重にwikiに登録しない（既存ページがあれば追記）。
- wikiページは **テキストのみ**。画像埋め込み (`![](...)`) は使わない。

## raw/ に置けるファイル

主に想定する形式：
- `.md` / `.txt` — Markdown / プレーンテキスト（推奨）
- `.html` — ブラウザのクリップなど
- `.pdf` — Claude Code は PDF を直接読める
- `.json` / `.csv` — 構造化テキスト

ファイル名の推奨: `YYYY-MM-DD_短いスラッグ.md`（例: `2026-05-12_data-pipeline.md`）。
日本語ファイル名も可。Windows禁止文字（`\ / : * ? " < > |`）は使わない。

## URLが含まれる場合の取り扱い（重要）

raw/ のテキストに URL が含まれている場合、AI は **WebFetch** ツールでその URL を取得し、内容を wiki に反映してください。

- 取得した URL の内容は、wiki 本文の参考情報として使う。
- 取得した URL を wiki ページのフロントマターに **`references:`** として記録する（オプション）。
- 認証必要・取得失敗の URL は、本文に「不明: URL取得失敗（理由）」として明示し、ページを完成させる。
- URL取得は **適度な範囲で**（無限再帰や明らかに大量取得になる場合は数件で切り上げる）。

例：
```markdown
---
sources: ["2026-05-12_note.md"]
references:
  - "https://note.com/yasuhitoo/n/naf7246ce43cc"
---
```

## wiki/ ページの構造

```markdown
---
title: ページタイトル（人間可読・日本語可）
aliases: [別名1, 別名2]                # 任意
tags: [タグ1, タグ2]                    # kebab-case 推奨
sources:                                # 出典 raw ファイル。複数可。ファイル名のみ
  - "2026-05-12_data-pipeline.md"
references:                             # 任意。WebFetchで参照したURL
  - "https://example.com/article"
created: 2026-05-12
updated: 2026-05-12
---

# ページタイトル

## 概要
（1〜3行の要約）

## 本文
（概念の詳しい説明。必要に応じて WebFetch で取得した参考情報を反映）

## 関連
- [[他のページタイトル]]
```

### フロントマターのルール
- `sources` は必須（meta タグページを除く）。raw のファイル名のみを記載。
- `references` は任意。WebFetchで参照した URL を列挙。
- `updated` はAIが編集するたびに更新する（今日の日付）。
- `created` は最初の作成日。上書きしない。
- `tags` は kebab-case 推奨。

## 粒度ルール

- 1つの raw ファイル → 複数の wiki ページが基本。
- 1ページの目安: 200〜800字程度。
- 極端に短い断片は独立ページにせず、関連ページに統合する。

## ファイル名規約（wiki/）

- 日本語可。Windows禁止文字は不可。
- `[[他のページ名]]` リンクは **ファイル名（拡張子なし）と一致** させる。

## /ingest 時の判断基準

新しい raw ファイルを処理するとき:

1. **対象列挙**: `raw/` 配下のファイルを Glob で取得（`.gitkeep` は除外）。
2. **既処理判定**: 各ファイル名について、`wiki/**/*.md` を Grep し、`sources:` に記載があれば **スキップ**。
3. **読込**: Read ツールで未処理ファイルを読む。
4. **URL抽出**: 本文中の URL（`https?://...`）を列挙し、各URLを **WebFetch** で取得して内容を把握。
5. **概念抽出**: テキスト + 取得した URL 内容から、主要な概念を抽出。
6. **既存ページ照合**: 各概念について `wiki/` の title/aliases/ファイル名・全文を Grep。
7. **編集**:
   - 既存ページあり → Edit で追記、`sources:` 追加、`updated:` 更新、必要なら `references:` 追加。
   - なし → Write で新規作成。
   - 矛盾発見 → `> **要レビュー:** ...` フラグを残す（自分で解決しない）。
8. **相互リンク**: 新規/更新ページから関連既存ページへ `[[link]]`、可能なら逆方向も。
9. **報告**: 処理結果を簡潔にまとめる。**raw/ のファイルは絶対に削除・改名しない**。

## やってはいけないこと

- `raw/` のファイルの削除・改名・編集
- 画像を埋め込むこと（`![](...)`）
- `sources:` 未記録のまま wiki に書くこと（処理済み判定が壊れる）
- WebFetch なしで「URL の内容はおそらく〜」と憶測で書くこと
- 既存ページの全面書き換え（追記・修正で対応）
- フロントマターの省略
- 確証のない断定（不明点は本文に「不明: 〜」と明示）

## /lint で検出すべき項目

- `sources:` が空のページ（meta タグ付きページは除外）
- `sources:` 記載のファイルが `raw/` に存在しないページ（dead source）
- どこからもリンクされていない孤立ページ
- title重複 / aliases重複
- 「要レビュー」フラグが残っているページ
- `raw/` にあるが、どの wiki ページの `sources:` にも記載がないファイル（処理漏れ）
- wiki ページ内の画像埋め込み `![](...)`（仕様違反）
